# AxiomWallet — macOS reference wallet

> **This is a reference example for SDK consumers, not a shipping
> product.** AxiomWallet.app demonstrates one way to build a desktop
> AXIOM wallet on top of the Rust SDK FFI. Real wallets, custodial
> backends, mobile clients, and gateways can all copy from here, but
> nobody is meant to install this binary as their daily-driver
> wallet. See `docs/AXIOM_DESIGN_MacOSReferenceApps.md`.

## Two-app pairing

AxiomWallet.app does **not** ship SMTP / IMAP / POP3. It writes UMP
envelopes to `<walletDir>/outbox/new/` and reads from
`<walletDir>/maildir/inbox/new/`. A separate app, **AxiomKiddo.app**
(`apps/macos/AxiomKiddo/`), watches those directories and bridges
to standard mail protocols. The split is the point — it makes the
CLAUDE.md §8 SDK ↔ transport boundary visually inescapable: the
filesystem layout between the two apps is the only IPC.

```
┌────────────────────┐   outbox/new/   ┌────────────────────┐
│  AxiomWallet.app   │ ──────────────▶ │  AxiomKiddo.app    │ ──▶ SMTP
│  (this app)        │                 │  (gateway)         │
│  TCP → Nabla only  │ ◀────────────── │                    │ ◀── POP3
└────────────────────┘   inbox/new/    └────────────────────┘
```

You need both running for sends / redeems / heals / genesis claims
to complete. The wallet alone will time out — that's deliberate, see
§6 of the design doc.

## Quick start

```bash
# 1. Bring up the dev env (in a separate terminal)
cd <axiom-root>/src
python3 scripts/axiom-env.py start

# 2. Build and run the wallet
cd apps/macos/AxiomWallet
./build.sh
swift run AxiomWallet

# 3. Build and run Kiddo (in a third terminal)
cd ../AxiomKiddo
./build.sh
swift run AxiomKiddo
```

In Kiddo, open Settings and add an account pointing at the wallet
directory (the wallet creates `~/Library/Application Support/Axiom/wallets/<pair>-normal/`
on first onboarding). Set the SMTP / POP3 hosts to the env's IP
(127.0.0.1 for solo dev, the box's LAN IP for cross-machine).

Once both apps are running and configured, the wallet's onboarding
genesis-claim should round-trip in 5-15 seconds.

## What this app demonstrates

1. **uniffi end-to-end.** Rust SDK → C ABI → Swift bindings, with
   the build pipeline (`build.sh`) regenerating bindings whenever
   the FFI surface changes.
2. **The §8 boundary in code.** Every cell of UI data comes from
   the FFI; the only network call the wallet makes is direct TCP
   to Nabla. SMTP/IMAP/POP3 belong to Kiddo, not here.
3. **Real wallet UX shape.** Onboarding, send/receive, activity,
   heal, diagnostic, key rotation, multi-pair (Normal + Ark).
   Production wallets would polish further; this is the minimum
   to soak-test the SDK end-to-end on macOS.

## Build outputs

`./build.sh` does:

1. `cargo build -p axiom-sdk-ffi --release` — Rust static library.
2. `uniffi-bindgen` — generates `Generated/AxiomSdk/axiom_sdk_ffi.swift`
   + `Generated/AxiomSdkFFI/{axiom_sdk_ffiFFI.h, module.modulemap}`.
3. `swift build` — compiles + links against the static lib.

Open the directory in Xcode for IDE work — it autodetects
`Package.swift`. Re-run `./build.sh` whenever you change the Rust
FFI surface (`sdk/ffi/src/lib.rs`).

## Headless smoke test

```bash
swift run SmokeTest
```

Same FFI surface, no UI — useful for CI / quick regression checks
without spinning up a windowed app.
